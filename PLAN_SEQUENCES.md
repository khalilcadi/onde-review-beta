## SECTION D — Comportement du cron `generate-actions`

### Références code
- [generate-actions/route.ts:210-228](app/api/crons/generate-actions/route.ts#L210-L228) — `checkStepCondition()` appelé sur le step **précédent**
- [generate-actions/route.ts:547-637](app/api/crons/generate-actions/route.ts#L547-L637) — fonction `checkStepCondition()`
- [generate-actions/route.ts:113-131](app/api/crons/generate-actions/route.ts#L113-L131) — `syncAcceptedInvitations()` au début du cron
- [generate-actions/route.ts:503-537](app/api/crons/generate-actions/route.ts#L503-L537) — fonction `isDelayReady()`

### Q1 — Step 1 = message (pas d'invitation), condition `if_connected` : le cron vérifie-t-il la condition avant d'envoyer ?

**Non.** Le cron vérifie la condition du **step précédent**, pas du step courant. Dans le code :

```ts
// Ligne 210-213
const previousStep = steps.find((s) => s.step_order === sl.current_step);
const conditionResult = await checkStepCondition(
  supabase, sl.lead_id, previousStep?.condition ?? null
);
```

Si step 1 est le **premier** step (donc `current_step=0`), `previousStep` = undefined → `condition = null` → `return "run"`. **Le message part direct**, sans vérifier le stage du lead.

👉 **Conclusion Q1 :** mettre une condition `if_connected` sur un step 1 = message **ne fonctionne PAS**. La condition n'est jamais évaluée.

### Q2 — Lead `stage=invited` dans séquence step 1 = message + condition `if_connected` : attend-il l'acceptation ?

**Non**, par la même raison que Q1. Le message partira dès que `delay_days` est écoulé, sans attendre que le lead soit connected.

**EN REVANCHE**, si la séquence est structurée comme :
- step_order=1 : `invitation` avec condition `if_connected` (dans ce step)
- step_order=2 : `message`

Alors :
- Au premier cycle, `current_step=0`, `nextStep=1` (invitation), pas de previousStep → condition null → envoie l'invitation.
- Après envoi, `current_step=1`.
- Au cycle suivant, `nextStep=2` (message), `previousStep=1` (invitation) → condition `if_connected` évaluée.
- Si `stage=invited` → `checkStepCondition` retourne **`"wait"`** (ligne 581) → le cron skip ce cycle SANS faire avancer.
- Quand `syncAcceptedInvitations` transitionne `invited → connected` (automatique au début de chaque cron), la condition devient vraie au cycle suivant → message envoyé.

👉 **Conclusion Q2 :** OUI, le cron attend l'acceptation **SI** la condition est sur le step d'invitation (qui est le `previousStep` du message). Le message part automatiquement quand le lead devient connected.

### Q3 — Skip/cancel ou wait ?

Le cron **WAIT** (pas skip). Du code `checkStepCondition()` :
- `if_connected` + stage non-connecté → `return "wait"` → skip ce cycle, retente au prochain (ligne 581)
- `if_no_response` + lead a répondu → `return "skip"` → avance le sequence_step sans action (ligne 621)
- `if_responded` + pas de réponse → `return "wait"` → idem
- `always` ou par défaut → `return "run"`

👉 **Conclusion Q3 :** pour `if_connected`, le cron attend. On **peut** mettre des leads `invited` dans une séquence, à condition que la structure soit correcte (invitation-placeholder en step 1, message en step 2 avec condition sur step 1).

### Bonus — Mécanisme d'auto-sync invited → connected

Ligne 113-131 : `syncAcceptedInvitations()` s'exécute au début de chaque cron. Il :
1. Interroge Unipile pour récupérer les nouvelles relations
2. Passe les leads de `invited` → `connected` si l'invitation a été acceptée
3. Le cron enchaîne avec la génération

👉 Donc l'enchaînement `invitation → wait → message` est **entièrement automatisé** dès lors que la structure de séquence est correcte.

## SECTION E — Proposition de séquences

### Structure type recommandée (basée sur Section D)

**Séquence "Invitation → M1 → M2"** (pour leads à partir de zéro) :
```
step 1 : invitation      delay=0        condition=if_connected
step 2 : message (M1)    delay=1-2j     condition=if_no_response
step 3 : message (M2)    delay=5-7j     (pas de condition, ou if_no_response)
```
- Step 1 attend l'acceptation avant d'envoyer M1 (grâce à condition sur step 1 qui gate step 2)
- Step 2 ne part que si le lead n'a pas déjà répondu (condition sur step 2 qui gate step 3)

**Séquence "M1 direct → M2"** (pour leads déjà connected) :
```
step 1 : message (M1)    delay=0        condition=if_no_response
step 2 : message (M2)    delay=5-7j     (pas de condition nécessaire, le if_no_response sur step 1 gère le skip)
```

- step 2 : `message` (M1), `delay_days=1`
- step 3 : `message` (M2), `delay_days=6`, `condition=if_no_response` (sur step 2)

**⚠️ PROBLÈME CRITIQUE** : ces leads ont **déjà** une invitation envoyée (stage=invited). Si on les met dans cette séquence avec `current_step=0`, le cron va **renvoyer une invitation** → duplicate, mauvais signal Unipile.

**Solution** : ajouter les sequence_leads avec `current_step=1` (le step invitation est virtuellement « déjà fait »). Le cron cherchera nextStep=2 (message), évaluera previousStep=1 (invitation) avec condition `if_connected`.

```sql
INSERT INTO sequence_leads (sequence_id, lead_id, current_step, status, entered_at)
VALUES ('<seq-id>', '<lead-id>', 1, 'active', NOW());
```

**Risque** : moyen. Si la condition est mal configurée ou si `syncAcceptedInvitations` ne transitionne pas le stage, les leads resteront bloqués indéfiniment. **Test obligatoire sur 2-3 leads avant import massif.**

#### Séquence 3 — `V2 Relance M2` (optionnel)
**But** : envoyer un M2 aux 5 leads qui ont reçu uniquement M1 sans réponse (Tableau C, msgs=1).

**Leads** : 5 (Ahmet, Alain BONNETAUD, Julian PERRIER, Anthony D., Franck LAMBOT)

⚠️ **Attention** : 2 sont HORS_ICP (Julian, Franck) → à retirer. Reste 3 leads pertinents (Ahmet-B, Alain-B, Anthony-D1).

**Structure** :
- step 1 : `message` (M2 relance), `delay_days=0`, `generation_mode=ai`, `condition=if_no_response`

**Insertion** : `current_step=0` (step 1 = M2 direct).

**Risque** : faible (3 leads), mais le prompt M2 doit être conscient que c'est une relance (déjà géré par le prompt : `messageStepNumber=2`, `isFirstContact=false` — MAIS attention : le cron calcule `messageStepNumber` à partir des messages déjà envoyés **dans la séquence**, pas globalement. Voir ligne 329 : `previousMessages.length === 0` → sera `isFirstContact=true` pour cette nouvelle séquence. **⚠️ Bug potentiel : le lead recevra un M1 au lieu d'un M2.**).

**Workaround** : utiliser `generation_mode=template` avec un template M2 codé en dur, OU modifier `buildUserPrompt` pour injecter l'historique global du lead.
