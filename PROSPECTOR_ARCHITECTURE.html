Patterns architecturaux

Server/Client Split

page.tsx (server component) fetch les données via Server Actions, puis passe en props à *-client.tsx (client component) qui gère l'UI et l'interactivité.

Hybrid API

Server Actions pour le CRUD (type-safe, avec cookies). Route Handlers pour IA (long-running), webhooks (service role), et crons (schedule).

supabaseOverride Pattern

Les fonctions partagées (callAI, getDecryptedApiKey, buildSystemPrompt) acceptent un client Supabase optionnel pour le contexte cron (pas de cookies).

Détail des étapes

A. Import → Pipeline

Sources d'import

  • CSV : Upload dans Pipeline, auto-détection séparateur (, ou ;), mapping 20+ variations de headers, anti-doublon sur linkedin_url
  • Manuel : Formulaire dans Pipeline (firstName, lastName, linkedinUrl requis)
  • Webhook : relation.created quand quelqu'un accepte une invitation

Anti-doublon

Chaque lead est identifié par son linkedin_url. Si l'URL existe déjà, l'import est rejeté (pas de mise à jour).

État initial

Tout nouveau lead entre avec : stage: to_invite, status: cold, score: 0.

B. Enrichissement + Scoring

Enrichissement (Perplexity + Unipile)

Route POST /api/ai/enrich — Deux sources combinées :

  • Unipile (LinkedIn) : profil complet, 5 derniers posts, expérience, date début poste
  • Perplexity (web) : news <3 mois, funding <18 mois, CA/taille, contexte secteur

Signal classifié en priorité : INBOUND → POST_DOULEUR → POST_SUJET → ACTUALITE → SIGNAL_FAIBLE → FROID

Auto-correction stage : si network_distance === "FIRST" et stage to_invite/invited → connected

Scoring IA (Claude/GPT)

Route POST /api/ai/score — Score 0-100 basé sur 3 axes :

AxePoints maxCritères
Fit Score40 ptsProfil solopreneur (10) + Secteur B2B (10) + Titre décisionnaire (10) + Signaux maturité ICP (10)
Intent Score40 ptsSignal base (0-20) + Bonus email/posts/tenure (0-15) + Bonus stage replied (+5)
Timing Score20 ptsOptimal (20) / Neutre (10) / Défavorable (0)

Catégories : HOT ≥70 WARM ≥45 COLD ≥25 NO_GO <25

C. Séquence → Actions quotidiennes

Cron generate-actions (06h00 Paris)

Parcourt toutes les séquences actives, pour chaque sequence_lead actif :

  1. Trouve le prochain step (current_step + 1)
  2. Vérifie delay_days écoulé depuis le dernier envoi
  3. Évalue la condition du step (always, if_connected, if_no_response, if_responded)
  4. Vérifie l'idempotence (pas d'action existante pour ce step)
  5. Vérifie le quota du jour
  6. Génère le message (template interpolé OU appel IA agent prospection)
  7. Crée l'action avec status: pending

Résultats des conditions

ConditionRésultatEffet
alwaysrunGénère l'action
if_connected (pas connecté)waitSkip, réessaye demain
if_connected (connecté)runGénère l'action
if_no_response (a répondu)skipSkip permanent, avance le step
if_no_response (pas de réponse)runGénère l'action
if_responded (a répondu)runGénère l'action
if_responded (pas de réponse)waitSkip, réessaye demain
D. Validation → Envoi

Validation par l'utilisateur

Dans Daily Actions, l'utilisateur voit les messages pending. Il peut :

  • Valider : pending → validated, calcul de scheduled_at
  • Éditer : modifie final_message, reste pending
  • Régénérer : appel IA pour un nouveau message
  • Annuler : pending → cancelled
  • Reporter : nouvelle date pour scheduled_at

Envoi par le cron (toutes les 2 min)

  1. Jitter aléatoire 0-30s (anti-fingerprint)
  2. Recovery des orphelins processing > 10min → validated
  3. Lock atomique : validated → processing (empêche double envoi)
  4. Pour chaque action : vérifie délai anti-détection, exécute via Unipile
  5. Succès : processing → sent + avance la séquence
  6. Erreur 429 : rate limit → stop toutes les actions user
  7. Erreur 5xx : échec + stop
  8. Autre erreur : échec + continue
E. Réponses → Inbox

Webhook message.received

  • Unipile envoie un webhook à chaque nouveau message
  • Message entrant (is_sender: false) :
  • → Upsert conversation dans DB
  • → Insert message avec direction: inbound
  • → Si le lead est en stage invited, connected ou in_sequence → stage passe à responded
  • → Si le lead est dans une séquence active → sequence_leads.status = responded

Webhook relation.created

  • Quand une invitation est acceptée
  • → Si lead en stage invited → passe à connected

Types de steps

🔍 Visit

Visite le profil LinkedIn du lead via getUserProfile(). Pas de message envoyé. Sert à signaler l'intérêt.

👋 Invitation

Envoie une demande de connexion via sendInvitation(). Message limité à 300 caractères. Stage: to_invite → invited.

💬 Message

Envoie un message dans la conversation via createChat() ou sendMessage(). Requiert que le lead soit connecté. Support fragments humanisés.

📧 InMail

Message LinkedIn premium, même logique que message mais sans nécessiter la connexion.

Algorithme de scheduling non-uniforme (burst pattern)

Le scheduling simule un comportement humain avec des bursts d'activité séparés par des pauses.

    Action 7 (visit)       :a7, 10:38, 1m

Algorithme

Pour chaque action à planifier :
  1. Si dans un burst (2-3 actions) :
     → Intervalle court : 2-3 min (intervalMin × 1.0-1.5)
  2. Entre les bursts :
     → Gap long : 24-48 min (intervalMax × 3-5)
  3. Plancher anti-détection :
     → Max(intervalle burst, délai anti-détection requis)
  4. Clamp aux heures actives :
     → Si overflow → reporter au prochain jour actif
  5. Jitter 0-15 min au début de chaque journée
Fragmentation humanisée des messages

40% des messages de 3+ phrases sont fragmentés en 2-3 parties, envoyées avec un délai de 12-25 secondes entre chaque.

Message original :
"Bonjour Jean. Je vous contacte au sujet de JARVIS. C'est une plateforme
pour les solopreneurs. Pouvez-vous y jeter un œil ?"

→ Fragment 1 : "Bonjour Jean. Je vous contacte au sujet de JARVIS."
   [12-25 sec de pause]
→ Fragment 2 : "c'est une plateforme pour les solopreneurs."
   [12-25 sec de pause]
→ Fragment 3 : "Pouvez-vous y jeter un œil ?"

Transformations subtiles :
- 25% de chance : minuscule en début de fragment
- 50% de chance : retrait du point final

Workflow quotidien type

06:00 — Cron generate-actions
Génération automatique des actions du jour (séquences actives → pending)
09:00 — User ouvre Daily Actions
Valide les messages, édite si nécessaire. Calcul des scheduled_at avec distribution non-uniforme.
09:00-19:00 — Cron send-actions (toutes les 2 min)
Envois automatiques respectant anti-détection : 15min entre messages, 1-3min visite→invitation.
~12:00 — Webhook message.received
Notification nouvelle réponse → traitement dans Inbox. Lead passe à responded.
18:00 — Récap Dashboard
KPIs mis à jour : actions envoyées, réponses, leads chauds, taux de réponse.

Les 4 agents IA

🎯 Agent Prospection v4.3

Usage : Génération messages LinkedIn (invitations, messages, InMails)

Output : Texte brut (300 chars invitations, 500 connectés)

Modèle : Claude Sonnet / GPT (user choice)

Templates : INBOUND, POST_DOULEUR, POST_SUJET, ACTUALITE, SIGNAL_FAIBLE, FROID

Blocs RAG : positionnement, icp, offres, messaging, objections, use_cases, pain_points (7 blocs)

📊 Agent Scoring v4.2

Usage : Qualification leads 0-100 (Fit + Intent + Timing)

Output : JSON strict (score, catégorie, breakdown)

Modèle : Claude Haiku (temp 0.3, déterministe)

Anti-hallucination : Données manquantes = contribution 0

Blocs RAG : positionnement, icp, pain_points, regles_decisionnelles (4 blocs)

🔍 Agent Enrichissement v4.3

Usage : Enrichissement via Unipile (LinkedIn) + Perplexity (web)

Output : JSON (company, person, signal, confidence)

Modèle : Perplexity Sonar Pro (facts only)

Sources : Profil LinkedIn, 5 derniers posts, news web <3 mois

Blocs RAG : positionnement, icp (2 blocs)

💬 Agent Conversational v4.3

Usage : Cockpit IA — reporting pipeline, analyse lead, brief call, correction message

Output : Prose (pas de JSON)

Modèle : Claude Sonnet / GPT

Modes : Reporting, Lead Analysis, Brief Call, Message Correction, Offer/Positioning

Hiérarchie des prompts

1. user_prompts (Supabase)     → Priorité max (override utilisateur)
2. PROMPTS_DEFAULTS (code)     → Fallback (lib/ai/prompts/defaults.ts)
3. '' (string vide)            → Si agent non trouvé

Prompt caching (Claude uniquement)

System Prompt = [
  { text: AGENT_PROMPT + RAG_CONTEXT,  cache_control: "ephemeral" },  ← CACHÉ
  { text: RUNTIME_CONTEXT }                                           ← DYNAMIQUE
]

Le prompt agent + RAG (statique) est caché, le contexte lead (dynamique) est recalculé à chaque appel. Économie ~90% sur les tokens d'entrée cachés.

Prompts personnalisables

Chaque utilisateur peut surcharger les prompts système des 4 agents dans Settings → Prompts. Les overrides sont stockés dans user_prompts. Un bouton "Reset to Default" supprime l'override et réactive le prompt code.

RAG éditable

Dans Settings → Connaissances, l'admin peut éditer le contenu des 14 blocs RAG. Les overrides sont stockés dans user_rag_data (JSON). Le mapping agent→blocs est visible en lecture seule.
